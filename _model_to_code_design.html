3. モデルとコードの対応づけ

プログラムがどのように動作するのか検討するときには、振る舞いのモデルを使います。
振る舞いのモデルを作成する前に、構造のモデルと振る舞いのモデルがどのようなコードとして実現されるのかについて、このチュートリアルの開発で用いる方式を決めておきましょう。
このことを「モデルとコードの対応づけ」または「実装方式の設計」と呼ぶことにします。

このような対応づけを「モデル変換ルール」と呼ぶこともあります。また、開発プロセスのなかでこの対応づけを検討する工程を「方式設計」と呼ぶ人もいます。

3.1. なぜ対応づけの検討が必要なのか

ソフトウェア開発でよく目にする事態として、設計と実装が乖離してしまい、頼れるのは動いているコードだけというものがあります。 このとき、多く人が「設計書がないから」「設計書が更新できていないから」と説明します。 このことについて、もう少し詳しく考えてみましょう。

ソフトウェアの開発に使う設計書には次のような2つの役割があります。

設計書の役割

  • 「何を作るか」について、構造と振る舞いをもって示す。

  • 「どのようにして作るか」について、設計書と実装の対応づけをもって示す。

「何を作るか」の設計書は、どんな機能を提供し、それがどのような動作・振る舞いとなっているかを示す設計書です。 例えば、「機能的な要求の発見・獲得→要求の分析→外部設計→内部設計」という開発プロセスがあるとき、「要求分析のユースケース図」、「外部設計のクラス図・シーケンス図」、「内部設計のクラス図・ステートマシン図」などを作成します。 これらが「何を作るか」を示すための設計書です。

では、「どのようにして作るか」の設計書とはどんなものでしょうか。 これは、例えば、設計書にクラスがあったらC言語の構造体に対応づけるとか、状態遷移は2次元配列を使って遷移表として実装しようとか…そういった、設計書に記載した要素に対して、実装するときにコードのどの部分にどのように反映するかを示した設計書です。

先に挙げた「何を作るか」の設計書には、「どのような実装方法で実装するのか」は書いてありません。 しかし、「どのようにして作るか」の設計書がないと、その側面は実装者がコードを作成しながら考えることになります。 そうなると、「どのようにして作るか」は、実装者ごとに考えることになり、また、同じ実装社でも実装する毎に異なってきます。しかも、その考えはコードに埋め込まれ、明文化されません。 設計書として明文化されていないならば、実装に不具合があって設計書に戻ったとしても、検討すべき情報がそこにはないということになります かくして、設計と実装の乖離ができあがるというわけです。

「モデル駆動開発（MDD）」という開発手法に、「モデル・タイプマッピング」と呼ばれる、2つのモデルの間の対応づけについての考え方があります。モデルとコードの対応づけは、タイプマッピングの一種とみなすこともできます。
この章の役割についてもう少し

この章で検討したいことは、ボウリングのスコアに関してどのようなモデルを作るのか、という話題から少し離れています。 モデルをどのように実装するのかということは、モデルの表し方とコードの表し方の間の関係を定めるものであって、ボウリングのスコアをどのようなモデルとして表すかということとは独立している話題だからです。

もし、この章の役割がピンとこないとか、実装についての議論にはあまり関心がないのであれば、最初はこの章を読み飛ばしてみてもかまいません。 後の章へ進むと、実装する人がモデルを使う際には、モデルとコードを対応づける必要があり、事前に実装方式を決めておかなければ、実装者が実装方法について考える必要があることに気づくでしょう。 そのときは、この章に戻って、どうして実装方式の設計が必要なのか考えてもらえたらと思います。

C言語でのクラスの表現

C言語はオブジェクト指向言語ではありませんが、構造体(struct)を使ってクラスの概念を表現できます。

  • クラス → 構造体(struct)

  • 属性 → 構造体のメンバー変数

  • メソッド → 構造体へのポインタを第1引数に取る関数

  • コンストラクタ → xxx_init() 関数

  • カプセル化 → ヘッダーファイルとソースファイルの分離、static関数

そして、実装ファイル`sample.c`は リスト 3.2 のようになります。

リスト 3.2 sample.c: 「Sample」クラスの実装（.cファイル）
#include "sample.h" 

/* Sample構造体の初期化（インスタンスの作成） */
void sample_init(Sample *sample) {  
    sample->attr_a = true;  
    sample->attr_b = true;  
    /* 状態の初期化は後で追加 */
}

/* playメソッドの実装 */
void sample_play(Sample *sample) {  
    /* 振る舞いは後で実装 */
    printf("play called attr_a:%d\n", sample->attr_a);
}

実装方式の策定も設計の一部

C言語に限ったことではありませんが、モデルとコードの対応づけ（実装方式）は、決まったものがあるわけではありません。重要なのは、自分たちがチームで決まった方式を決めておき、モデルの作成（設計時）にもその実装方式を前提に設計し、実装時にもその実装方式に従って実装することです。

たとえば、「テスト駆動開発による組み込みプログラミング」 [embtdd] では、「SOLIDの原則をCモジュールに提要した設計モデル」として次のようなモデルを紹介しています。

シングルインスタンスモジュール（Single-instance module）

モジュールの内部構造をカプセル化するが、モジュールのインスタンスがひとつであることを前提とする。

マルチインスタンスモジュール（Multi-instance module）

モジュールの内部構造をカプセル化して、モジュールのインスタンスを複数作れるようにする。

動的インターフェイス（Dynamic interface）

モジュールのインターフェイス関数を実行時に割り当てられるようにする。

型による動的インターフェイス（Pre-type dynamic module）

同じインターフェイスを持つモジュールの型が複数あり、それぞれが独自のインターフェイス関数を持てるようにする。

この分類からみると、このチュートリアルが採用した実装方式は、「マルチインスタンスモジュール」に相当することがわかるでしょう。

さらに、クラス、シーケンス、状態遷移といった設計上の概念を、実装時にどのように実現するのかを決めておくこともまた重要な設計上の決定になります。設計の方針ともなるので、「実装方式の決定はアーキテクチャ設計である」と捉える場合もあります。

3.2.5. 状態とイベントと状態遷移の関係

状態とイベント

ステートマシン図において、状態は「イベント」の発生を待っているところです。 多くの場合、イベントは、そのクラス自身の操作では処理を先に進められなくなるようなできごとです。 たとえば、外部からの入力（操作待ちや受信待ちなど）や、一定の時間経過などがイベントの候補になります。

状態遷移

状態遷移は、ある状態において待っていたイベントが発生して、別の状態へ移ることを表しています。 イベントには、イベントが発生したとき、実際に遷移するか判定する条件を追加できます。 この条件のことを「ガード条件」と呼びます。 ガード条件つきのイベントでは、イベントが発生したときにガード条件を評価し、条件が真なら状態遷移します。 ガード条件が偽のとき、イベントは起きたことになります（消費されると呼びます）が、状態は遷移しません。

そして、状態遷移には、イベントが発生したときに実行したい処理を追加できます。 この処理のことを「アクション（またはエフェクト）」と呼びます。

イベントが発生すると、ガード条件つきのイベントならさらにガード条件も真なら、状態遷移に伴うアクションが実行され、それから次の状態へ遷移します。

3.2.6. ステートマシン図にイベントやアクションを追加する

まず、「ST0」から「ST1」への状態遷移を、イベントが「ev1」でガード条件は「gd（が真）」のとき、アクション「act1」を実行するよう編集してみましょう。

状態遷移にイベントやアクションを追加する（１）

  1. 「ST0」から「ST1」への状態遷移を選択し、この遷移のプロパティーを表示し、ベースタブを開く。

  2. プロパティーを編集する（ 図 3.13 ）。

    • 「トリガー」にイベント名を設定する。ここでは「ev1」というイベントを設定する。

    • 「ガード」にガード条件を設定する。ここでは「gd1（が真）」を設定しする。

    • 「アクション」にアクションを設定する。ここでは「act1」という処理があるとして、これを設定する。

      • act1は、2つの引数（イベントとパラメーター）を持つメソッドと想定する。

図 3.13 「ST0」から「ST1」への状態遷移のイベントとアクションを編集する

次に、「ST1」からの「ST2」への遷移です。 この遷移に割り当てるイベントは、「ev2」、アクションを「act2」とします。 そして、このアクションの実行後、ガード条件「gd2」が真なら「ST2」へ、偽ならアクション「act3」を実行してから「ST1」へ遷移するように変更してみます。

遷移先が2つあるなら、状態遷移を別々に引けばよさそうです。 ところが、同じイベントを待ってる遷移先が2つ以上あると、どちらの状態へ遷移するのか決定できなくなります（あいまいな状態遷移と呼びます）。 そこで、1つのイベントによる遷移先が状況によって変わる場合には「選択疑似状態」を使います。

3.3.7. 状態ごとの処理関数

各状態での処理は、イベントに応じて適切な処理を行います。

リスト 3.10 sample.c: 状態「ST0」での処理
static void st0_proc(Sample *sample, SampleEvent evt, int param) {
    switch (evt) {  
        case EV1:  
            if (gd1(sample)) {  
                act1(sample, evt, param);  
                sample->state = ST1;  
            } else {
                printf("Event ignored due to guard condition\n");  
            }
            break;
        case EV3:  
            act3(sample, evt, param);
            sample->state = ST0;
            break;
        default:
            /* その他のイベントは無視 */
            break;
    }
}
イベントで場合分けする
イベントが「EV1」の場合
ガード条件による判定
アクション「act1」の呼び出し
次の状態「ST1」への遷移
ガード条件が偽だったときの処理
イベントが「EV3」の場合
C言語での関数のスコープ

C言語では、`static`キーワードを使って関数のスコープを制限できます：

  • `static`のない関数: 他のファイルから呼び出し可能（public相当）

  • `static`付き関数: 同じファイル内からのみ呼び出し可能（private相当）

`st0_proc`などの内部処理関数は`static`にして、外部から直接呼び出せないようにします。

  end

  def play
    # 略
  end

  def gd1?

  end

  def gd2?

  end
end

3.3.9. 状態遷移を追加する（１）

状態ごとの処理を受け持つメソッドを作成します。 メソッド名は、状態名を小文字して「_proc」をつけるというルールにします。 したがって、状態名「ST0」の場合、メソッド名は「st0_proc」になります（ リスト 3.12 ）。

リスト 3.12 【Ruby】「ST0」の状態遷移のメソッドを作成する
class Sample
  def initialize
    # 略
  end

  def play
    # 略
  end

  def st0_proc(evt, param)
    case evt 
    when :ev1 
      if gd1? 
        puts "    gd1: #{gd1?}"
        act1(evt, param) 
        @state = :ST1 
      else 
        puts "    <<< gd1: #{gd1?}, transition is ignored. >>>"
      end
    when :ev3 
      act3(evt, param)
      @state = :ST2
    end
  end

  # gd1?, gd2? が続く
end

次の状態「ST1」への遷移。
ガード条件が偽だったとき。イベントが無視されたことを表示する。

3.5. まとめ

クラス図とステートマシン図で動作を表現すれば、Rubyのコードに変換できることがわかりました。

3.5.1. 構造のモデルを作成した手順

構造のモデルに登場する構成要素や要素間の関連を見つけ出し、スコアシートの構造を整理しました。

スコアシートの構造のモデルを作成した手順

  1. スコアシートに関する構成要素の発見

    • 具体的なスコアシートを観察して、そこにある要素を洗い出してオブジェクト図で表した。

  2. スコアシートに関するクラス図の作成

    • オブジェクト図に登場したオブジェクトからクラスを、オブジェクト同士のリンクから関連を見出し、クラス図に表した。

3.5.2. 構造のモデルだけではプログラムは作れない

構造のモデルを作ったことで、プログラムを構成する要素と、要素同士の関係（関連）を表せるようになりました。 また、クラスのインスタンスをオブジェクト図に見合うように作成すれば、実際に観察したスコアシートと同じようなデータ構造を作成できることもわかりました。

ところで、作りたかったのは、ボウリングのゲームスコアをつけるプログラムでした。 つまり、プレーヤーがフレームごとに交代して投球し、そのたびにスコアを更新する処理です。 ですが、これまでに作成した構造のモデルでは、ゲームの進行に応じてどんなことを処理するか表せていません。 そのためには、処理の中身を表す「振る舞いのモデル」を作成する必要があります。 次の章では、ステートマシン図を使って、フレームやスコアの振る舞いを表すモデルを作成します。
